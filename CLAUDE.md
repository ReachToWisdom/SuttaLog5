## 프리즘 글로벌 규칙
(prism의 list_rules에서 자동 동기화 — 모든 AI 에이전트 공통)

### ai:goal-driven-execution

목표 주도 실행 — 성공 기준 정의 후 검증까지 루프.
작업을 검증 가능한 목표로 변환:
- "validation 추가" → "잘못된 입력 테스트 작성 후 통과시킴"
- "버그 수정" → "버그 재현 테스트 작성 후 통과시킴"
- "X 리팩터" → "전후 모두 테스트 통과 보장"
다단계 작업 시 짧은 계획 진술:
1. [단계] → 검증: [확인]
2. [단계] → 검증: [확인]
강한 성공 기준 → 독립 루프 가능.
약한 기준 ("동작하게 하라") → 끊임없는 명확화 필요.

### ai:idempotent-migrations

마이그레이션 SQL 멱등성 — INSERT 추가 시 ON CONFLICT / OR IGNORE / OR REPLACE 명시.
재실행해도 동일 결과 보장. 중복 누적 방지.

### ai:simplicity-first

단순함 우선 — 문제를 푸는 최소 코드. 사변적 추가 X.
- 요청하지 않은 기능 추가 X.
- 1회용 코드의 추상화 X.
- 요청하지 않은 "유연성" 또는 "설정성" X.
- 불가능한 시나리오에 대한 에러 처리 X.
- 200줄을 50줄로 쓸 수 있으면 다시 작성.
- 자문: "시니어 엔지니어가 이걸 과복잡하다고 할까?" Yes면 단순화.

### ai:surgical-changes

수술적 변경 — 필요한 곳만 건드림. 자기가 만든 것만 정리.
기존 코드 편집 시:
- 인접 코드/주석/포맷팅 "개선" X.
- 부서지지 않은 것 리팩터 X.
- 본인 스타일과 다르더라도 기존 스타일 유지.
- 무관한 죽은 코드 발견 시 — 삭제 말고 *언급*만.
변경이 고아를 만들 때:
- 본인 변경이 만든 unused import/var/fn은 제거.
- 기존 죽은 코드는 요청 없으면 제거 X.
테스트: 변경된 모든 줄이 사용자 요청에 직접 추적 가능해야.

### ai:task-decomposition

큰 작업은 검증 가능한 최소 sub-task로 분할.
- 모듈 분할/큰 리팩터: 단계별 (신설 → 호출부 → 테스트), 각 단계 단독 빌드 통과.
- 200줄 초과 변경 또는 4KB 초과 Edit이 예상되면 자동 분할.
- TaskCreate로 sub-task 등록 + 단계별 in_progress→completed.
- 각 sub-task 후 빌드+회귀 검증. 다음 sub-task 시작 전 회귀 0 확인.
- 사용자가 명시적으로 큰 작업 요청해도 안전을 위해 sub-task로 묶어 진행.

### ai:tdd-discipline

TDD 규율 — 동작 변경엔 검증 동반.
- 새 동작 추가: 통과 케이스 + 실패 케이스 1개 이상 단위 테스트.
- 버그 수정: 재현 테스트 먼저 (red), 수정 후 통과(green).
- 리팩터: 기존 테스트가 있어야 진행. 없으면 특성화 테스트 먼저.
- 순수 함수/파서/룰 평가기 등 분리 가능한 단위는 항상 테스트 동반.
권장 패턴: `#[cfg(test)] mod xxx_tests` 같은 파일 또는 별도 파일.

### ai:think-before-coding

구현 전 사고 — 가정·불확실성·트레이드오프를 명시.
- 가정은 명시적으로 진술. 불확실하면 질문.
- 다중 해석 가능 시 모두 제시 — 묻지 말고 침묵 선택 X.
- 더 단순한 접근이 있으면 말함. 정당하면 사용자 의견에 반대도.
- 무언가 불명확하면 멈춤. 무엇이 헷갈리는지 명명. 질문.
